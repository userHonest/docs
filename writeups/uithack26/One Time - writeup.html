UiTHack26 Radio Intercept Crypto Writeup

This challenge presents a short fictional military radio intercept, a block of hexadecimal ciphertext, and one important hint hidden in the dialogue itself: the sender complains that the key is tiny and specifically mentions Kasiski. That is enough to narrow the problem from “unknown ciphertext blob” to “short repeating-key polyalphabetic encryption,” and from there the solve is straightforward with a crib-based attack.

Category: Cryptography Theme: Reused short key Technique: Repeating-key XOR Solve: Crib dragging with known flag prefix
Recovered flag: UiTHack26{4_r3us3d_k3y_1s_0nly_wh3n_us3d_0n_d1ff3r3nt_m3ss4g3s_r1ght?}

Immediate Observations

  • The transmitted block is valid hexadecimal and has even length, so it can be interpreted as raw bytes.
  • The operators complain about a very short key.
  • The explicit reference to Kasiski is the real hint. Kasiski examination is associated with repeated-key polyalphabetic ciphers such as Vigenere.
  • In modern CTF form, the simplest byte-oriented analogue is repeating-key XOR.
  • The expected flag format is almost certainly UiTHack26{...}, which gives us a known plaintext prefix.

Step 1: Normalize the Ciphertext

The first step is to join the hex lines into one continuous ciphertext string and confirm the size. I used a small Python snippet for that. 

ct_hex = (

ct = bytes.fromhex(ct_hex)
print(len(ct))

The output is:

70

So the transmission is 70 bytes long. That is short, but still more than enough to test repeated-key hypotheses.

Step 2: Extract the Real Hint

The dialogue is doing almost all the cryptanalytic work for us. It tells us:

  • the key is reused, because it is “the same garbage as last week,”
  • the key is short, because it fits on a matchbook,
  • the speaker explicitly knows Kasiski examination, which is used to attack repeated-key ciphers.

That does not prove the cipher is classical Vigenere over letters. It only strongly suggests a repeated key. Because the ciphertext is given as hex, the cleanest working hypothesis is:

The plaintext was encrypted with a short repeating byte key using XOR, then the resulting bytes were shown in hexadecimal.

Before going directly to the right answer, I still checked a few alternative models. That is good practice, because challenge authors sometimes layer encodings or encrypt hex text instead of raw bytes.

Step 3: Check Repetition Signals

Classical Kasiski-style attacks look for repeated substrings. On a 70-byte ciphertext we should not expect huge repeated sequences, but quick triage is still useful. 

ct = bytes.fromhex(ct_hex)

    seen = {}
    reps = []

Step 4: Estimate Plausible Key Lengths

A common first pass against repeating-key XOR is to measure similarity statistics across key lengths. I used an index-of-coincidence style heuristic over byte columns. 

ct = bytes.fromhex(ct_hex)

for k in range(1, 17):
    cols = [ct[i::k] for i in range(k)]
    iocs = []
    for col in cols:
        n = len(col)
        freqs = {b: col.count(b) for b in set(col)}
        ic = sum(v * (v - 1) for v in freqs.values()) / (n * (n - 1)) if n > 1 else 0
        iocs.append(ic)
    print(k, round(sum(iocs) / len(iocs), 4))

Step 5: Try Generic Single-Byte Column Solvers

If this is repeating-key XOR on English text, then for a guessed key length we can split the ciphertext into columns and solve each column as a single-byte XOR problem. I tried that as a sanity check. 

import string

ct = bytes.fromhex(ct_hex)
common = set(bytes(
    string.ascii_letters + string.digits + string.punctuation + " \\n\\r\\t",
    "ascii"
))

def score(pt):
    total = 0
    for b in pt:
        if b in common:
            total += 2
        elif 32 <= b < 127:
            total += 1
        else:
            total -= 5
    return total

for k in range(1, 13):
    key = []
    for i in range(k):
        col = ct[i::k]
        best = None
        for kb in range(256):
            pt = bytes(c ^ kb for c in col)
            sc = score(pt)
            if best is None or sc > best[0]:
                best = (sc, kb, pt)
        key.append(best[1])

    pt = bytes(c ^ key[i % k] for i, c in enumerate(ct))
    print("k", k, "score", score(pt), "key", bytes(key).hex(), "repr", repr(pt[:80]))

This actually gave a notable result at k = 10. The best-scoring key from generic printable-text scoring was:

and the corresponding plaintext was:

'y,287

That is not the flag. It is merely printable garbage. This is a useful failure: it tells us generic English scoring is not reliable enough here because the true plaintext is a leetspeak flag, not normal prose. The challenge is easier to solve with a crib than with unsupervised scoring.

Step 6: Use the Known Flag Prefix as a Crib

At this point the best move is not to keep guessing. In CTFs, the flag format is usually known, and here the prefix is almost certainly UiTHack26{. If the plaintext starts with that string, then the first ten plaintext bytes immediately reveal the ten-byte key: 

ct = bytes.fromhex(ct_hex)
known = b"UiTHack26{"
key = bytes(c ^ p for c, p in zip(ct[:len(known)], known))
print(key.hex())

The output is:

e3982dc5b0f8c0ca87ee

This is exactly the kind of key we would expect if the scheme is repeating-key XOR over raw bytes: a short arbitrary byte string, not a human-readable password.

Now decrypt the full ciphertext with that candidate key. 

pt = bytes(c ^ key[i % len(key)] for i, c in enumerate(ct))
print(pt.decode())

That yields the complete flag immediately:

UiTHack26{4_r3us3d_k3y_1s_0nly_wh3n_us3d_0n_d1ff3r3nt_m3ss4g3s_r1ght?}

Once that plaintext appears, the solve is done. The output is perfectly structured, uses the expected flag wrapper, and is semantically coherent with the challenge theme.

Why the Attack Works

Repeating-key XOR encrypts each plaintext byte with a key byte chosen by position modulo the key length: 

cipher[i] = plaintext[i] XOR key[i mod key_length]

XOR is reversible, so if we know both the ciphertext byte and the plaintext byte for any position, we recover the corresponding key byte: 

key[i mod key_length] = cipher[i] XOR plaintext[i]

Because the flag prefix is ten bytes long and the real key length is also ten bytes, the known prefix recovers the entire key in one shot. That is why this challenge collapses instantly once the correct model is chosen.

If the key length had been longer than the known prefix, this would only have recovered part of the key, and we would have needed more crib dragging or statistical recovery. Here we got lucky because the challenge was designed to be clean and direct.

Manual Walkthrough Without Code

  1. Read the dialogue and notice the explicit Kasiski reference.
  2. Infer that the intended weakness is a short repeating key.
  3. Interpret the transmission as raw bytes encoded in hex.
  4. Assume the plaintext is a flag beginning with UiTHack26{.
  5. XOR the first ten ciphertext bytes with those ten known plaintext bytes.
  6. Recover the full ten-byte key e3982dc5b0f8c0ca87ee.
  7. Repeat that key across the entire ciphertext and XOR again.
  8. Read the resulting plaintext flag.

This is one of those challenges where the right interpretation matters far more than brute force. Once the hint is understood correctly, the solve is almost immediate.
