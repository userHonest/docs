Binary Picture CTF Writeup

This challenge hides the flag inside the visible pixel structure of text.png. The image looks like a decorative arrangement of colored blocks, but the blocks are actually a binary transport format. Once the geometry is measured correctly, the picture decodes into readable ASCII text and reveals the flag.

UiTHack26{ken_designed_utf8}

Overview

The solve path was not traditional metadata steganography. The PNG metadata was ordinary, and a simple strings pass did not expose the flag. The useful clue came from the image itself: it was clearly arranged on a fixed-size grid with a very small palette.

What mattered

  • The picture is aligned to a regular 20x20 cell size.
  • Gray cells are separators, not data.
  • White cells represent empty bits.
  • Every non-white, non-gray cell can be treated as a filled bit.

What did not matter

  • The exact foreground color of a filled cell.
  • EXIF metadata or hidden text chunks.
  • OCR on the rendered shapes as if they were ordinary letters.

The palette looks meaningful at first, but it is mostly visual styling. The payload is encoded by occupancy, not by hue.

Reasoning

After opening the image, the repeating separators suggested that the file was built from a grid of block cells. Counting the dimensions showed the image reduced cleanly into 62 columns by 64 rows of 20x20 tiles. The light gray rows and columns formed dividers between groups.

The key breakthrough was to stop looking for hidden bytes inside the PNG container and instead interpret the visible grid as raw binary data. If a cell is white, it becomes 0. If a cell is colored, it becomes 1. Gray is a separator and should not be treated as part of the byte stream.

Each horizontal band contains 8 data rows, which strongly matches the height of a byte. Reading each data column from top to bottom yields one 8-bit value. Converting those bytes to ASCII reconstructs plaintext. Once that direction was tested, the output immediately began with: 

According to all known laws of aviation, there is no way a bee should be able to fly...

That confirmed the decoding model. Continuing the extraction produced the embedded flag inside the recovered text stream: 

...flies anyway because bees UiTHack26{ken_designed_utf8} don't care...

Exact extraction code

The following Python script is the complete decoder. It loads the image, reduces it to 20-pixel cells, finds the gray separators, reads each 8-cell-tall data column as a byte, and prints the ASCII result. 

from PIL import Image

img = Image.open("text.png").convert("RGB")
block_size = 20

WHITE = (255, 255, 255)
GRAY = (230, 230, 230)

# Reduce the full image into a coarse grid where each entry represents one 20x20 tile.
grid = []
for y in range(0, img.size[1], block_size):
    row = []
    for x in range(0, img.size[0], block_size):
        pixel = img.getpixel((x, y))
        if pixel == GRAY:
            row.append("#")
        elif pixel == WHITE:
            row.append(".")
        else:
            row.append("1")
    grid.append("".join(row))

# The visible image is arranged in 7 bands of 8 data rows each, separated by gray rows.
bands = []
y = 0
while y < len(grid):
    if set(grid[y]) == {"#"}:
        y += 1
        continue
    start = y
    while y < len(grid) and set(grid[y]) != {"#"}:
        y += 1
    bands.append((start, y - 1))

decoded = []

for y0, y1 in bands:
    x = 0
    while x < len(grid[0]):
        column = "".join(grid[yy][x] for yy in range(y0, y1 + 1))

        # A full gray column is a separator between groups.
        if set(column) == {"#"}:
            decoded.append("|")
            x += 1
            continue

        bits = "".join("1" if grid[yy][x] == "1" else "0" for yy in range(y0, y1 + 1))
        value = int(bits, 2)

        if 32 <= value < 127:
            decoded.append(chr(value))
        else:
            decoded.append(".")

        x += 1

print("".join(decoded))

Useful helper scripts from the solve

I used a couple of short helper snippets while reasoning through the image. They are not required once the byte-column model is known, but they show how the structure was verified.

1. Count the palette

from PIL import Image
from collections import Counter

img = Image.open("text.png").convert("RGB")
counts = Counter(img.getdata())

for color, amount in counts.most_common():
    print(color, amount)

2. Locate full gray separator rows

from PIL import Image

img = Image.open("text.png").convert("RGB")
pix = img.load()

GRAY = (230, 230, 230)
WHITE = (255, 255, 255)

separator_rows = []
for y in range(img.size[1]):
    colors = {pix[x, y] for x in range(img.size[0])}
    if colors <= {GRAY, WHITE}:
        separator_rows.append(y)

print(separator_rows)

Why OCR was a dead end

It is tempting to run OCR because the shapes look vaguely typographic, but that approach is not robust here. The glyphs are not letters in the ordinary visual sense. They are grouped bytes. OCR produces noisy guesses because it tries to read the rendered blocks as Latin characters rather than decode the actual data layout.

The reliable method is structural decoding:

  • Measure tile size.
  • Classify cells into gray separator, white empty, or filled bit.
  • Extract 8-bit columns within each band.
  • Map each byte to ASCII.
