Rewind Reversing Challenge Writeup

This writeup covers the full static analysis of the rewind binary. The challenge is a straightforward reversing task: find the accepted key by understanding how the program transforms user input and comparing that transformed state against a hardcoded target table.

Category: Reversing Binary: rewind Platform: ELF 64-bit x86-64 PIE Approach: Static reverse + transform inversion
Recovered flag: UiTHack26{r3w1nd_th3_t4p3}

First Impression

This binary is very friendly to static reversing:

  • it is not stripped,
  • it contains debug info,
  • there is a visible main symbol,
  • the challenge logic lives directly inside that function.

Step 4: Disassemble main

The core command was:

objdump -d -Mintel /home/userhonest/uithack26/rewind/rewind

The disassembly is long, but the structure is very regular. The function does five things:

  1. initialize a 64-entry constant table on the stack,
  2. read user input with fgets,
  3. copy the first 64 input bytes into an integer array,
  4. apply several reversible transformations,
  5. compare the final result against a hardcoded 64-entry target array.

Input Handling

The binary prompts the user and reads up to 64 bytes:

malloc(0x40)
printf("C:\\> ENTER KEY:")
fgets(buf, 0x40, stdin)

Then it copies each input byte into a 32-bit integer array. This is important because all later operations happen on 32-bit integers, but the values are still just ASCII byte values.

Visible Transform Pattern

Once the input has been copied into the working array, the disassembly shows a very clean transformation chain. Each loop is easy to identify because the compiler emitted separate loop counters and temporaries for every stage.

Step 5: Reconstruct the Algorithm

Reading the loops in order gives this exact logic:

  1. Swap adjacent pairs: [0,1] [2,3] [4,5] ...
  2. XOR each element with a 64-entry constant table.
  3. Reverse the full 64-element array.
  4. Swap adjacent pairs again.
  5. Rotate the array by 7 positions.
  6. Compare against a hardcoded 64-entry target table.

In pseudocode, the logic is: 

read input[64]
work[i] = (signed char)input[i] for i in 0..63

for (i = 0; i <= 62; i += 2)
    swap(work[i], work[i + 1]);

for (i = 0; i <= 63; i++)
    work[i] ^= table1[i];

for (i = 0; i <= 31; i++)
    swap(work[i], work[63 - i]);

for (i = 0; i <= 62; i += 2)
    swap(work[i], work[i + 1]);

for (i = 0; i <= 63; i++)
    rotated[(i + 7) % 64] = work[i];

work = rotated;

if (work == target)
    puts("REWIND COMPLETE");
else
    puts("TAPE ERROR");
The key observation is that every one of these operations is reversible. That means we do not need brute force, symbolic execution, or dynamic instrumentation. We can just invert the pipeline and recover the original accepted input directly.

Step 7: Invert the Transformations

To recover the original input, we reverse the operations in the opposite order:

  1. Undo the rotate-by-7.
  2. Undo the second adjacent-pair swap.
  3. Undo the array reversal.
  4. Undo the XOR with the same table.
  5. Undo the first adjacent-pair swap.

Since swap, reverse, and XOR are all self-inverse, only the rotation requires careful indexing.

# Invert rotation:
# forward: out[(i + 7) % 64] = work[i]
# inverse: work[i] = out[(i + 7) % 64]
a = [target[(i + 7) % 64] for i in range(64)]

# Undo second pair swap
for i in range(0, 64, 2):
    a[i], a[i + 1] = a[i + 1], a[i]

# Undo reverse

# Undo first pair swap
for i in range(0, 64, 2):

Subtle Implementation Detail

One detail worth noting is that the binary compares all 64 integer slots, not just the visible flag length. The recovered state after inversion is:

  • the flag text,
  • followed by a newline byte,
  • followed by null bytes for the rest of the 64-byte state.

In a perfect implementation, the buffer would be explicitly initialized or the code would compare only the meaningful input length. Here, the intended accepted input is still obvious, but the implementation leaks that the author expected a standard line input ending in \n.

Why This Works Well as a Challenge

This is a clean reversing exercise because it rewards reading control flow rather than throwing tools at the problem blindly. The transformations are simple enough to recognize in assembly:

  • pair swaps show up as two loads and two stores,
  • XOR is visible directly,
  • reverse is recognizable through symmetric index access,
  • rotation appears as indexed writes using modular arithmetic.

Once the transformation chain is reconstructed, the solve is deterministic and elegant.
