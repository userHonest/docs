Base 179 Challenge Writeup

A full writeup for the challenge prompt: I've invented a new encoding algorithm: base 179! Can you decode this message? The core trick is that the challenge does not actually start with a real “base 179” encoding. Instead, it uses a layered encoding chain and relies on the fake label to push solvers toward the wrong decoding strategy.

1. Reading the Hint Correctly

The challenge says: 

I've invented a new encoding algorithm: base 179! Can you decode this message?

That statement is the entire trap. The natural reaction is to try to interpret the ciphertext directly as some unfamiliar positional number system with radix 179. But the given string already contains strong evidence that this is not the right first step.

The ciphertext is: 

GFQTKTLXMZLXMUTQHBITS2LQN5UWQVKSJ5IEQUTRNJCHQZKUNUZVUSSTGVCFGZ3EIM4FCQ32LA3TKZKIMNDE2T3QMVVVGMJSI4======

There are two immediate clues:

  1. It only uses characters from the Base32 alphabet.
  2. It ends in ======, which is classic Base32 padding.

So before trying anything exotic, the correct move is to test the obvious standard encodings. That alone unwraps the first layer.

In encoding challenges, the “special new algorithm” is often not the outermost layer. If the ciphertext visibly matches a standard format, decode that first.

2. Stage One: Base32

Decoding the original string as Base32 gives: 

1a5MwfWvRp8Q9ipoihUROPHRqjDxeTm3ZJS5DSgdC8QCzX75eHcFMOpekS12G

This output is still not the flag, but it is clearly printable and structured. That usually means there is another text-oriented encoding stage inside it.

3. Why Base179 Still Looks Wrong Here

At this point, the challenge hint starts to interfere with good judgment. A solver can spend a lot of time trying to force the stage-one string into some hand-made base-179 interpretation:

  • treating ASCII bytes as digits in base 179
  • subtracting character offsets to make values fall into 0..178
  • trying big-endian and little-endian integer reconstruction
  • trying affine transforms or XOR/addition with 179

None of those paths gave a clean, trustworthy UiTHack{...} output. More importantly, the stage-one text still looks like a plausible standard-base payload, which suggests we should continue peeling ordinary encodings instead of forcing the hint too early.

The next good heuristic is this: the stage-one output is entirely alphanumeric, so it is a valid candidate for Base62.

4. Stage Two: Base62

The string: 

1a5MwfWvRp8Q9ipoihUROPHRqjDxeTm3ZJS5DSgdC8QCzX75eHcFMOpekS12G

decodes cleanly as Base62 using the alphabet: 

0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz

That yields: 

<G>E'@:O)"1N$\VA2.t7?SOEF0Qo6m?TDIPBJ=RT1i?iP

This intermediate output is significant for two reasons:

  1. It is fully printable.
  2. It contains punctuation patterns that are typical of ASCII85 / Base85 style encodings.

Base62 Decoder

import string

stage1 = "1a5MwfWvRp8Q9ipoihUROPHRqjDxeTm3ZJS5DSgdC8QCzX75eHcFMOpekS12G"
alphabet = string.digits + string.ascii_uppercase + string.ascii_lowercase

n = 0
for ch in stage1:
    n = n * 62 + alphabet.index(ch)

out = bytearray()
while n:
    n, r = divmod(n, 256)
    out.append(r)

stage2 = bytes(out[::-1]).decode()
print(stage2)

Output: 

<G>E'@:O)"1N$\VA2.t7?SOEF0Qo6m?TDIPBJ=RT1i?iP

6. Full Solve Script

The following Python script performs the entire decode chain end to end: 

import base64
import string

cipher = "GFQTKTLXMZLXMUTQHBITS2LQN5UWQVKSJ5IEQUTRNJCHQZKUNUZVUSSTGVCFGZ3EIM4FCQ32LA3TKZKIMNDE2T3QMVVVGMJSI4======"

# Stage 1: Base32
stage1 = base64.b32decode(cipher).decode()
print("Stage 1:", stage1)

# Stage 2: Base62
alphabet = string.digits + string.ascii_uppercase + string.ascii_lowercase
n = 0
for ch in stage1:
    n = n * 62 + alphabet.index(ch)

buf = bytearray()
while n:
    n, r = divmod(n, 256)
    buf.append(r)

stage2 = bytes(buf[::-1]).decode()
print("Stage 2:", stage2)

# Stage 3: ASCII85
flag = base64.a85decode(stage2).decode()
print("Flag:", flag)

7. What “Base 179” Was Really Doing

The challenge title and wording were mostly misdirection. The string did not begin in a custom base-179 representation at all. Instead, the challenge relied on the solver overcommitting to the stated hint and missing the visible structure of the ciphertext.

In practice, the lesson is:

  1. Always test obvious standard encodings first.
  2. Do not trust the challenge description more than the data itself.
  3. When each decoding stage produces clean printable output, continue following the structure.

The phrase “base 179” may simply be part of the joke or thematic framing, while the actual solve path is a chain of familiar encodings: Base32, then Base62, then ASCII85.
