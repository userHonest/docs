Recovering the Flag From a Broken PNG

This challenge looked simple at first glance: there was a single PNG file and the task was to recover a flag in the format UiTHack26{...}. The catch was that the PNG was malformed, so ordinary image viewers and straightforward string searches were not enough to solve it cleanly.

Recovered flag: UiTHack26{wh4t_d0_pn6_574nd_f0r_4nyw4y5}
File analyzed
challenge.png
Core issue
PNG header claimed 0 x 0 dimensions
Actual approach
Parse chunks, decompress IDAT, reconstruct pixels manually

Initial Recon

The workspace contained one obvious target: challenge.png. There was also a copy with a token in the filename and a Windows Zone.Identifier sidecar, but nothing else suggested an alternate path to the flag.

The first useful check was the file type: 

file challenge.png
challenge.png: PNG image data, 0 x 0, 8-bit/color RGB, non-interlaced

A PNG reporting width and height as zero is immediately suspicious. It means the container is structurally recognizable as a PNG, but the header is not valid for a normal image. That strongly suggests one of two things:

  1. The file is intentionally corrupted to prevent direct viewing.
  2. The image data is still present and can be recovered by parsing the file at a lower level.

Running strings did not reveal the flag directly. That was expected. If the flag were hidden as plain text in metadata, the challenge would have ended there. Instead, the file mostly contained compressed image data.

Understanding the PNG Structure

PNG files are chunk-based. They begin with a fixed signature, then a series of chunks such as:

  • IHDR for image width, height, bit depth, and color type
  • IDAT for compressed pixel data
  • IEND to terminate the file

A quick hex dump confirmed that the file really was a PNG and that the width and height fields in IHDR had both been set to zero: 

00000000: 89 50 4e 47 0d 0a 1a 0a 00 00 00 0d 49 48 44 52
00000010: 00 00 00 00 00 00 00 00 08 02 00 00 00 3e 0f 5f
00000020: ad 00 00 80 00 49 44 41 54 78 da ...

That told us something important: the image payload had not been removed. It started immediately after the header in a large set of IDAT chunks. So the challenge was not about metadata extraction. It was about recovering the real image from a deliberately invalid header.

Extracting the Raw Image Stream

The next step was to parse all PNG chunks, concatenate every IDAT payload, and decompress the result with zlib. A short Python script was enough for that: 

import struct, zlib

b = open("challenge.png", "rb").read()
o = 8
chunks = []

while o < len(b):
    length = struct.unpack(">I", b[o:o+4])[0]
    o += 4
    chunk_type = b[o:o+4]
    o += 4
    data = b[o:o+length]
    o += length
    crc = b[o:o+4]
    o += 4
    chunks.append((chunk_type, data))
    if chunk_type == b"IEND":
        break

idat = b"".join(data for chunk_type, data in chunks if chunk_type == b"IDAT")
raw = zlib.decompress(idat)
print(len(raw))

The decompressed stream length was 3240900 bytes. That is far too structured to be random. For an 8-bit RGB PNG, each row normally has:

  • 1 filter byte
  • width * 3 bytes of RGB pixel data

That means the raw size must satisfy: 

raw_size = height * (1 + width * 3)

Solving for plausible width and height pairs yielded a handful of mathematical candidates, but only one realistic image resolution: 

1200 x 900

That was a strong signal that the original image dimensions had likely been 1200x900 before the header was sabotaged.

Reconstructing the Image Properly

At that point the file still could not be opened directly, because simply changing width and height is not enough. PNG scanlines are filtered before compression, and the decoder must reverse those filters row by row.

The decompressed stream showed that the image used multiple PNG filter types, including:

  • 1 (Sub)
  • 2 (Up)
  • 3 (Average)
  • 4 (Paeth)

Because of that, a correct reconstruction needed a manual decoder that reverses each filter using the previous row and neighboring pixels. The logic looked like this: 

def paeth(a, b, c):
    p = a + b - c
    pa = abs(p - a)
    pb = abs(p - b)
    pc = abs(p - c)
    if pa <= pb and pa <= pc:
        return a
    if pb <= pc:
        return b
    return c

# For each row:
# read filter byte
# unfilter row according to PNG rules
# append final RGB bytes

Once the filters were reversed and the RGB bytes were assembled into a new image buffer, the reconstructed output became a valid PNG again. The recovered image showed a man in a suit holding a glowing screen, with faint green text repeated across the entire image.

The visible text was not hidden in file metadata. It was embedded visually inside the pixel content, and the corrupted header was the mechanism used to stop normal viewers from displaying it.

Reading the Flag

After reconstruction, the repeated green overlay was readable by eye, and OCR confirmed the full string. Because the text repeated many times over the image, some OCR attempts produced small casing and spacing mistakes, but the intended flag was still clear. 

UiTHack26{wh4t_d0_pn6_574nd_f0r_4nyw4y5}

The joke in the flag is the use of pn6 and leetspeak substitutions, echoing the question “what do PNG stand for anyways”.

Why This Worked

The challenge relied on a common CTF pattern: break just enough of the file format to defeat standard tools while leaving the interesting data intact. In this case:

  • The PNG signature remained valid.
  • The IDAT stream remained intact and decompressible.
  • Only the image dimensions were corrupted in IHDR.

That makes the challenge approachable if you know how PNG files are structured. Once the raw image stream is extracted, the rest is a matter of finding the correct dimensions and applying the PNG filter rules correctly.

Short Solve Path

  1. Identify that challenge.png is malformed because it reports 0 x 0 dimensions.
  2. Parse all PNG chunks and concatenate the IDAT data.
  3. Decompress the combined IDAT stream with zlib.
  4. Infer plausible dimensions from the decompressed size.
  5. Choose the realistic candidate 1200x900.
  6. Reverse the PNG scanline filters for every row.
  7. Save the recovered pixel buffer as a new valid image.
  8. Read the repeated text overlay and extract the flag.

Final Answer

The flag is: 

UiTHack26{wh4t_d0_pn6_574nd_f0r_4nyw4y5}
