Challenge Summary

fax_machine is a Linux x86_64 pwn challenge built around a broken run-length decoder. The fax machine accepts image dimensions and a compressed stream of black and white pixel runs. The hidden twist is that the decoder does not stop writing once the destination image is full. Because of that, the final RLE token can keep writing beyond the heap buffer and corrupt adjacent heap metadata.

The binary is also generous enough to leak a heap address and a libc address before asking for input. That combination makes the challenge much less about brute force and much more about recognizing the intended heap technique.

Architecture

x86_64 ELF

Bundled glibc

libc-2.27.so

Main Technique

House of Force

Hidden Target

win() prints flag.txt

Bottom line: the challenge is solvable with a top-chunk size overwrite, followed by a controlled allocation into libc, followed by a hook overwrite.

Initial Recon and Runtime Setup

The challenge ships with three important files:

  • fax_machine
  • ld-2.27.so
  • libc-2.27.so

That strongly suggests the intended runtime is the bundled glibc 2.27, not the system libc. For accurate offsets and stable exploitation, the binary should be launched with the provided loader and library path.

./ld-2.27.so --library-path . ./fax_machine

A useful convenience step is creating a symlink so the old loader finds the expected soname:

ln -s libc-2.27.so libc.so.6

The binary itself is not stripped, which makes reversing much easier. Important symbols are visible directly, including rle_decode, fax_machine, banner, and win.

Program Logic

At a high level, the binary behaves like this:

  1. main() prints the intro banner.
  2. banner() leaks a heap pointer and the address of free().
  3. fax_machine() enters a loop asking for width and height.
  4. It calls rle_decode(width, height) to allocate and fill a bitmap.
  5. If the bitmap size is at most 0x1000 bytes, it prints the image and frees it.
  6. If the bitmap is larger than 0x1000, it skips free() and loops back for another image.

This size gate is critical. The exploit intentionally uses a large bitmap in stage 1 so the corrupted heap state stays alive. If the bitmap were freed immediately, the exploit would become much more fragile.

RLE Input Format

The program decodes a text-based run-length encoding. Each token is:

  • W<count> for a run of white pixels
  • B<count> for a run of black pixels

The decoder accepts uppercase or lowercase color letters. Tokens are separated by spaces and read from one input line using fgets().

For example:

W8 B8 W8 B8

would generate alternating blocks of 8 white and 8 black bits.

The challenge hint, “manually compress the image data,” is really telling us to stop thinking like a normal user who wants to send a picture and start thinking like an attacker who wants to craft the decoder’s write pattern directly.

The Vulnerability in rle_decode()

The decoder computes the total number of bits as width * height, allocates exactly enough bytes for those bits, and then parses each RLE token one by one.

The bug is not in the allocation size itself. The bug is in the inner write loop. Once a run length has been parsed, the decoder writes that many bits into the output buffer without checking, inside the loop, whether the bitmap is already full.

total_bits = width * height
total_bytes = (total_bits + 7) >> 3
buf = malloc(total_bytes)

for each token:
    parse color and count
    for i in range(count):
        write one bit to buf[current_bits / 8]
        current_bits += 1

# the bound check happens only outside the per-bit loop

That means the final token can be oversized and continue writing past the end of the chunk. The last run is the dangerous one because the outer loop will stop only after the decoder finishes the entire token.

Exploit idea: choose a bitmap size that places the top chunk immediately after the destination buffer, then make the final run large enough to smash the top chunk size field.

Heap Layout and Why the Top Chunk Is Reachable

The decoder uses one heap allocation for the bitmap. For a fresh process, that allocation comes from the main arena, and the next chunk after it is normally the top chunk. So if we write far enough past the bitmap, we do not need a complex heap grooming strategy. We can directly corrupt the top chunk metadata.

The exploit uses a width and height that decode to 0x1001 bytes:

width = 4097
height = 8
total_bits = 4097 * 8 = 32776
total_bytes = (32776 + 7) >> 3 = 4097 = 0x1001

This size is ideal for two reasons:

  1. it is large enough that fax_machine() will skip the later free(),
  2. it still gives a predictable top-chunk position relative to the leaked heap address.

The stage-1 overflow token is:

B33024

That run is much larger than the declared image size and keeps writing into the next chunk’s metadata, including the top chunk size field.

Banner Leaks and Why They Matter

The binary gives away exactly the two addresses needed for a clean libc heap exploit:

  • a heap leak from malloc(1)
  • a libc leak from the resolved address of free()

Those leaks let us compute:

  • the position of the current top chunk, and
  • the location of hooks and gadgets inside libc.

House of Force Math

Once the top chunk size is corrupted to a huge value, the next allocation can be aimed at a target address by choosing the right request size.

The exploit uses the leaked heap pointer to estimate the top chunk after stage 1:

old_top = heap + 0x1020

That value matches the chunk alignment and metadata placement for the 0x1001-byte allocation. From there, if we want the next user pointer to become target_user, the request size is:

req = ((target_user - 0x10) - old_top) - 8

This is the standard House-of-Force relation:

  • old_top points to the current top chunk header,
  • target_user - 0x10 converts a desired user pointer back to a chunk header,
  • - 8 compensates for how glibc rounds and accounts for request sizes.

Primary Exploit: __free_hook -> win()

Stage 1: Create the top-chunk overwrite

The exploit sends:

width  = 4097
height = 8
RLE    = B33024

This allocates a bitmap of 0x1001 bytes, overflows into the top chunk, and leaves the bitmap allocated because the program skips freeing large images.

Stage 2: Move the top chunk near __free_hook

Using the libc leak, the exploit computes:

libc_base  = free_addr - 0x80490
free_hook  = libc_base + 0x3af8c8
stage3_user = free_hook - 0x28

The -0x28 landing is not arbitrary. It was found to be a stable offset that allows a forward write into __free_hook without depending on unstable exact-adjacent chunk placement.

Stage 3: Overwrite the hook

The final stage writes:

payload = b"\\x00" * 40 + p64(win)

Because the stage-3 user pointer is at __free_hook - 0x28, writing 40 bytes of filler positions the win pointer exactly over __free_hook.

Trigger

After the stage-3 bitmap is decoded, the program follows its normal logic for small images: it prints the image and then calls free(bitmap). glibc checks __free_hook before doing the real free logic, so control transfers directly to win().

__free_hook(bitmap, caller) -> win()

Extra arguments are harmless on x86_64 here because win() does not care about them.

Result: the primary solver is stable and verified. It prints the flag if the runtime environment has a real flag.txt.

Alternate Exploit: __free_hook -> setcontext+0x35 -> win()

The second exploit intentionally uses a different control-flow sink. Instead of pointing __free_hook straight at win(), it points it at a useful gadget inside libc: setcontext+0x35.

Why setcontext+0x35 is useful

In glibc 2.27, starting at this offset, setcontext restores a collection of registers from memory pointed to by rdi, sets rsp from a field in that structure, pushes a value loaded from another field, and then returns.

The important restored fields are:

Offset from fake frame Register / use
+0xa0 restored rsp
+0xa8 loaded into rcx, then pushed before ret
+0x68 restored rdi

That means the freed chunk can be turned into a fake frame. Once free() calls our overwritten hook, setcontext+0x35 uses the chunk contents as execution state.

How the alternate payload works

The stage-3 chunk still lands at __free_hook - 0x28. Its content is now structured like this:

[0x00 .. 0x27]  filler
[0x28 .. 0x2f]  __free_hook = setcontext+0x35
...
[0xa0 .. 0xa7]  fake rsp = stage3_user + 0x208
[0xa8 .. 0xaf]  fake rcx = win
[0x208 ..]      tiny stack area containing win

When free() fires:

  1. glibc calls __free_hook
  2. control enters setcontext+0x35
  3. rsp is restored from our fake frame
  4. rcx is pushed
  5. ret transfers execution into win()

This is a genuine second exploit path because it uses the same heap primitive but a different control-transfer mechanism. Instead of “hook directly to win,” it performs “hook to context restore, pivot to fake state, then jump to win.”

Result: the alternate solver solve_setcontext.py was also verified locally with a temporary test flag.txt.

Diagram 4. Alternate exploit path with setcontext+0x35

What Was Tried and Rejected

Several other routes were explored but not validated for this binary/runtime pair:

  • __malloc_hook overwrite
  • _IO_list_all / fake FILE structure
  • exit callback / atexit structure corruption
  • __free_hook -> system
  • __free_hook -> puts
  • common glibc 2.27 one-gadgets

Those ideas were reasonable, but they were not stable here. The direct win() path and the setcontext pivot were the two methods that actually held up under testing.

What the transcript means

  1. heap @ ... is the leaked heap address from the banner.
  2. free() @ ... is the leaked libc address used to compute libc base and hook addresses.
  3. Receiving 4097x8 fax... is stage 1, the large bitmap used to corrupt the top chunk.
  4. Receiving 140598309275736x8 fax... is the huge House-of-Force allocation request used to move the top chunk to libc.
  5. Receiving 384x1 fax... is the small stage-3 allocation that overwrites the hook.
  6. The long B...W... line is the RLE encoding of the stage-3 payload.
  7. The decoded fax image is just the program rendering the payload bits as a monochrome image before it calls free().

So that output is not random noise. It is the visible proof that the exploit progressed through all three stages and delivered the final payload.

Final Notes

There are two verified exploit files in the workspace:

  • solve.py - the primary __free_hook -> win() exploit
  • solve_setcontext.py - the alternate setcontext pivot exploit

Recovered flag: UiTHack26{5p1771n6_f4x_1n70_7h3_w1ld3rn355}

Both were verified locally using temporary test flag.txt files. The real challenge flag is not present in the current local workspace, so the final real flag only appears if the challenge environment contains the intended flag.txt.

Most important takeaway: this challenge is a clean example of how a single missing bounds check in a decoder can be transformed into a top-chunk overwrite, then into a libc hook overwrite, and finally into reliable code execution.
