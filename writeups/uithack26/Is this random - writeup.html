Writeup: Weak RSA / Legacy SSH Key Challenge

This report documents the offline analysis of a recovered SSH public key from a legacy Debian-era system, including fingerprinting, modulus extraction, tool-assisted analysis, and validation of the recovered cryptographic material.

Prepared as a technical challenge writeup in HTML format.

Scope note: This document is limited to local cryptographic analysis and validation of the provided key material. It intentionally does not include steps for authenticating to a live external host.

1. Challenge Context

The supplied prompt described an SSH public key recovered from a decommissioned Debian Etch server that went offline in 2009. That description strongly suggests historical key-generation weakness, especially the Debian OpenSSL entropy flaw from CVE-2008-0166. That was the initial hypothesis, but the actual recovery path still needed to be verified empirically.

6. Historical Hypothesis Testing

Because the challenge referenced Debian Etch, the first analytical branch was to test the common weak-Debian-key hypothesis. A local copy of the public Debian weak-key repository was obtained and examined. 

git clone https://github.com/g0tmi1k/debian-ssh
cd debian-ssh
grep -R -F "$(cut -d' ' -f2 ../given_key.pub)" .
cd ..

Direct matching of the raw SSH blob did not produce an immediate hit. That does not prove the hypothesis false, but it does mean the next practical step is to test the RSA key mathematically rather than continue with filename-based guessing.

7. Tooling Setup with RsaCtfTool

A dedicated virtual environment was created and RsaCtfTool was installed to run multiple RSA recovery strategies against the public key. 

git clone https://github.com/RsaCtfTool/RsaCtfTool
cd RsaCtfTool
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
pip install .

There were two minor setup issues during this stage:

  • An initial attempt to run python3 RsaCtfTool.py failed because the repository layout uses the modern package structure under src/, not the old single-file entrypoint.
  • An initial attempt to run python3 -m src.RsaCtfTool also failed because that package is not directly executable as a module entrypoint in this checkout.

Installing the project with pip install . resolved that and exposed the correct RsaCtfTool command-line entrypoint.

8. Early Environment Checks

Before focusing entirely on RSA factorization, several common legacy-key routes were checked. These checks were useful for narrowing the problem and ruling out misleading assumptions from the challenge text. 

ssh-vulnkey ../given_key.pub
grep -R "00:02:d5:af:29:27:6c:95:a4:9d:c2:ab:3b:50:67:07" /usr/share/ssh/

Those checks did not produce a direct usable result in the local environment. In particular, ssh-vulnkey was not present, and the local system did not provide a ready-made blacklist path that immediately mapped the fingerprint. That increased confidence that the fastest path was direct mathematical analysis of the key itself.

9. Broad Attack Evaluation

The public key was first tested across the full default attack set. Most attacks did not apply, but the key result was that the factordb attack succeeded. 

RsaCtfTool --publickey ../given_key.pub --attack all

Observed result excerpt:

[*] Testing key ../given_key.pub.
[*] Performing factordb attack on ../given_key.pub.
[*] Attack success with factordb method !

The broader attack run also showed that several standard categories such as small-prime checks, non-RSA detection, and common toy-CTF prime databases did not trigger. That further isolated the successful path to factor lookup.

10. Recovery Path: Factorable RSA Modulus

This is the decisive technical result. The recovery path was not merely a historical Debian weak-key lookup. The RSA modulus itself was recoverable through factorization data. Once the factors p and q are known, the private exponent d can be computed, and the full private key can be reconstructed. 

RsaCtfTool --publickey ../given_key.pub --attack factordb --private

That command yielded a PEM-encoded RSA private key. The full private key is intentionally omitted from this report.

11. Private-Key Validation

After saving the recovered PEM material to id_rsa, the corresponding public key was derived and compared against the original artifact. 

ssh-keygen -y -f id_rsa > derived.pub
diff -u derived.pub ../given_key.pub

Observed diff:

--- derived.pub
+++ ../given_key.pub
@@ -1 +1 @@
-ssh-rsa AAAA...==
+ssh-rsa AAAA...== root@targetcluster

The only difference was the trailing SSH comment field, root@targetcluster. That field is metadata and does not affect the cryptographic identity of the key. The base64-encoded key material matched.

12. Troubleshooting and Corrections

Several operational issues came up during the workflow, and each is worth documenting because they are common when turning recovered key material into a usable local artifact.

  • A relative-path mistake occurred when comparing derived.pub to given_key.pub from inside the RsaCtfTool directory. The fix was simply to compare against ../given_key.pub or run the comparison from the parent directory.
  • One saved copy of id_rsa became corrupted when challenge text was accidentally pasted into the PEM block. That copy had to be replaced with a clean version before any OpenSSH tooling would accept it.
  • The public/private comparison initially appeared different because the recovered public key generated by ssh-keygen -y did not include the original comment. This was not a cryptographic mismatch, only a metadata difference.

The working comparison used in the parent directory was: 

diff -u RsaCtfTool/derived.pub given_key.pub

14. Technical Interpretation

The challenge narrative suggested a legacy Debian OpenSSL bug, but the experimentally confirmed result is more direct: the modulus was factorable. That alone is enough to completely break RSA, regardless of how the key was originally generated.

  • The public key reveals the RSA modulus n and public exponent e.
  • If n can be factored into p and q, then phi(n) is known.
  • The private exponent d is computed as the modular inverse of e modulo phi(n).
  • The full RSA private key can then be reconstructed and exported in standard PEM format.

15. Why the Debian Hypothesis Was Plausible but Secondary

The Debian Etch framing was still a useful lead. A 2009-era Debian system immediately raises the possibility of the historical OpenSSL entropy bug, and that is exactly why the weak-key archive was checked first. However, once direct factorization-based recovery succeeded, the narrative hint became secondary to the concrete cryptographic result. From an analyst's perspective, the key point is not which story appears in the prompt, but which attack path is demonstrably successful against the actual modulus.

16. Defensive Lessons

  • Do not assume a challenge hint identifies the real cryptographic weakness; validate with tooling.
  • Any factorable RSA modulus must be treated as fully compromised.
  • SSH comments such as root@targetcluster are informational only.
  • Legacy keys should be audited against weak-key datasets and factorization sources during incident response.

18. Final Conclusion

Offline analysis shows that the supplied SSH public key corresponds to an RSA keypair whose private key can be reconstructed because the modulus is not secure. The reconstructed private key was validated locally by deriving the corresponding public key and confirming that it matched the original artifact.

19. Challenge Outcome

After recovering and validating the RSA private key, the challenge was considered solved. The reconstructed key matched the supplied public key and represented the credential intended by the challenge design.

This confirmed that the weakness was sufficient to recover the correct private key from the public material.
