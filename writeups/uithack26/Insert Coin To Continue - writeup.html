Insert Coin Challenge Write-up

Challenge prompt:

"You've found an old arcade machine with a mysterious binary. It asks for a password, but maybe you don't need to know it... Sometimes the answer is hiding in plain sight."

Category

Reverse Engineering / Basic Binary Inspection

Difficulty

Beginner

Core Idea

The flag is stored directly as a printable string inside the binary.

1. Initial Observation

The wording of the prompt is the main clue. It says the program asks for a password, but also hints that maybe the password is not necessary and that the answer may be "hiding in plain sight." In reversing challenges, that usually suggests:

  • The binary may contain useful embedded strings.
  • The password and flag might both be hardcoded.
  • A simple static inspection may be enough without disassembly or debugging.

The fastest first move is therefore to inspect the files and extract printable strings.

2. Files Present

Listing the directory showed two main binaries: 

strings
strings-arm

There were also Windows metadata files ending in :Zone.Identifier, which are not relevant to solving the challenge.

3. Identify the Binary Types

Running file revealed the platform differences: 

strings:     ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped
strings-arm: Mach-O 64-bit arm64 executable

This tells us:

  • strings is the Linux x86-64 binary.
  • strings-arm is a macOS ARM binary.
  • Both likely implement the same challenge logic on different architectures.

4. Extract Printable Strings

The key step was to run the Unix strings utility against the binaries. For the Linux binary, the relevant output included: 

arcade
UiTHack26{str1ngs_4r3_fun}
Usage: %s <password>
*** ACCESS GRANTED ***
FLAG: %s
*** ACCESS DENIED ***
HINT: Sometimes the answer is hiding in plain sight...

This already exposes almost everything:

  • arcade appears to be the expected password.
  • UiTHack26{str1ngs_4r3_fun} is clearly the flag format.
  • The hint strongly confirms that direct string extraction is the intended solution path.

The ARM binary contained the same meaningful strings, which further validated the conclusion.

5. Why This Works

Many beginner reversing challenges hardcode sensitive values in global variables or read-only data sections. If a binary is not packed, encrypted, or obfuscated, the strings utility can reveal human-readable data directly from sections such as:

  • .rodata in ELF binaries
  • __cstring in Mach-O binaries

Since this executable was not stripped in a way that removed string literals, both the password and the flag were visible without any need for:

  • disassembly
  • debugging
  • patching
  • brute force

6. Runtime Verification

To confirm the finding, the Linux binary was executed with the discovered password. The original file did not have execute permission, so a copy was placed in /tmp, marked executable, and run: 

cp strings /tmp/strings
chmod +x /tmp/strings
/tmp/strings arcade

Program output: 

*** ACCESS GRANTED ***
FLAG: UiTHack26{str1ngs_4r3_fun}

This confirms both the password and the extracted flag.

7. Final Answer

The challenge can be solved entirely through static inspection of embedded strings.

UiTHack26{str1ngs_4r3_fun}

8. Short Conclusion

The intended lesson is straightforward: always check the easy path first. When a challenge hints that something is visible "in plain sight," extracting strings from the binary is often enough to recover secrets that were carelessly embedded by the developer.

In this case, the executable exposed both the password arcade and the flag UiTHack26{str1ngs_4r3_fun} directly in its printable data.
