Initial Setup and First Impressions

The challenge directory is small, which makes the target clear. The only files that matter are: 

model.pt
model.pt:Zone.Identifier
snAIke.py
snAIke.py:Zone.Identifier

The Windows Zone.Identifier files are irrelevant. The challenge description already points at the weights, so the natural plan is to inspect the model file instead of overthinking the game logic.

The first question is whether the checkpoint leaks anything obvious. Even if it does not, PyTorch checkpoints still expose raw tensor storage, so the right workflow is:

  • Read the model definition.
  • Work out how the checkpoint is structured.
  • Look at the raw bytes, not just the interpreted numbers.
The model only needs to look believable. It does not need to be a clean training artifact.

Reading the Snake Code

The Python file is short and gives the exact network architecture. That tells us what tensor shapes to expect in the checkpoint. 

class SnakeCNN(nn.Module):
    def __init__(self, h=GRID_H, w=GRID_W):
        super().__init__()
        self.conv = nn.Sequential(
            nn.Conv2d(1, 32, kernel_size=3, padding=1),
            nn.ReLU(),
            nn.Conv2d(32, 64, kernel_size=3, padding=1),
            nn.ReLU(),
            nn.Conv2d(64, 64, kernel_size=3, stride=2, padding=1),
            nn.ReLU(),
        )
        with torch.no_grad():
            dummy    = torch.zeros(1, 1, h, w)
            conv_out = self.conv(dummy).flatten(1).shape[1]
        self.head = nn.Sequential(
            nn.Linear(conv_out, 256),
            nn.ReLU(),
            nn.Linear(256, 4),
        )

From this alone we can derive the expected parameter layout:

  • conv.0.weight: 32 x 1 x 3 x 3 = 288 float32 values
  • conv.0.bias: 32 values
  • conv.2.weight: 64 x 32 x 3 x 3 = 18432 values
  • conv.2.bias: 64 values
  • conv.4.weight: 64 x 64 x 3 x 3 = 36864 values
  • conv.4.bias: 64 values
  • head.0.weight: 256 x 2304 = 589824 values
  • head.0.bias: 256 values
  • head.2.weight: 4 x 256 = 1024 values
  • head.2.bias: 4 values

The game loads the checkpoint using: 

ckpt = torch.load(args.model, map_location=device, weights_only=True)
if isinstance(ckpt, dict) and "weights" in ckpt:
    model.load_state_dict(ckpt["weights"])
else:
    model.load_state_dict(ckpt)

This suggests the checkpoint is a simple dictionary with a state dict under "weights", which makes low-level inspection straightforward.

What Might Be Hidden and Where

Once the tensors are mapped, the plausible hiding strategies are:

  1. Store the flag directly in spare metadata or training fields.
  2. Encode the flag in the decimal values of some weights.
  3. Use low-order bits of the float32 representation as a covert channel.
  4. Arrange a tensor so that, when reshaped as an image, it reveals a QR code or text.

The metadata route did not expose anything useful, so the likely answer was hidden in the tensor values or their bit patterns.

Float32 values are stored as raw 32-bit words, and tiny changes in the least significant bits usually do not affect model behavior much. That makes LSBs a natural hiding place.

That is enough to justify testing bit extraction directly.

Bit-Level Extraction of the Payload

The decisive step is to stop treating the storage as floats and read it as unsigned 32-bit integers instead. If one bit is hidden in each weight word, extraction is simple:

  1. Read a tensor blob as little-endian uint32.
  2. Take the least significant bit of each word.
  3. Pack those bits into bytes.
  4. Interpret the result as ASCII.

I tested this on each tensor. The immediate hit came from model/data/2, which maps to conv.2.weight. 

import zipfile
import numpy as np

zf = zipfile.ZipFile("model.pt")
arr = np.frombuffer(zf.read("model/data/2"), dtype="<u4")
bits = ((arr >> 0) & 1).astype(np.uint8)
msg = np.packbits(bits[:(len(bits) // 8) * 8], bitorder="big").tobytes()
print(msg[:80])

The beginning of the output was: 

b'UiTHack26{7h3_5n4k3_m1gh7_n33d_4_l4rg3r_m0d3l}\x00...'

That is the full flag, followed by a null terminator.

This works because conv.2.weight has 18432 float32 values, which gives 2304 bytes of capacity at one bit per weight.

Why big-endian bit packing matters

Bit order matters too. Using numpy.packbits(..., bitorder="big") produces the readable ASCII stream here.

So the exact decoding rule is: 

one bit from each 32-bit weight word
read in storage order
pack into bytes most-significant-bit first

Why this is a realistic hiding method

Flipping the least significant bit of a float32 rarely changes the value enough to matter for a simple model like this. That makes a checkpoint a convenient steganographic container:

  • large capacity
  • high entropy appearance
  • small perturbations tolerated by the model
  • most solvers will initially trust the checkpoint as purely numeric data

Clean Solver Script

Here is a minimal solver that extracts the flag without PyTorch: 

import zipfile
import numpy as np

with zipfile.ZipFile("model.pt") as zf:
    data = zf.read("model/data/2")

arr = np.frombuffer(data, dtype="<u4")
bits = ((arr >> 0) & 1).astype(np.uint8)
payload = np.packbits(bits[:(len(bits) // 8) * 8], bitorder="big").tobytes()
flag = payload.split(b"\x00", 1)[0].decode("ascii")

print(flag)

Running that prints: 

UiTHack26{7h3_5n4k3_m1gh7_n33d_4_l4rg3r_m0d3l}

More defensive version

A more general version can brute force all tensor blobs, bit positions, and bit orders, then scan for a flag pattern. 

import re
import zipfile
import numpy as np

with zipfile.ZipFile("model.pt") as zf:
    for name in zf.namelist():
        if not name.startswith("model/data/"):
            continue
        arr = np.frombuffer(zf.read(name), dtype="<u4")
        for bit in range(32):
            bits = ((arr >> bit) & 1).astype(np.uint8)
            n = (len(bits) // 8) * 8
            for order in ("big", "little"):
                data = np.packbits(bits[:n], bitorder=order).tobytes()
                m = re.search(rb"UiTHack26\{[^}]+\}", data)
                if m:
                    print(name, bit, order, m.group().decode())

That more generic scanner lands on model/data/2, bit 0, order big.

Takeaways

This is a good reminder that model files are arbitrary binary containers, not inherently trustworthy numeric artifacts.

  • Neural network weights can carry hidden data without breaking the model.
  • PyTorch checkpoints are often easy to inspect with generic archive and pickle tooling.
  • When the challenge text points at the weights, believe it.
  • Low-order bits are the first place to look when the visible values seem legitimate.

In practical reversing terms, the solve flow was:

  1. Read the model architecture from the Python source.
  2. Inspect the checkpoint as a zip archive.
  3. Map storage blobs to named tensors through data.pkl.
  4. Test raw-bit extraction on the weight blobs.
  5. Recover the flag from the LSB stream of conv.2.weight.
